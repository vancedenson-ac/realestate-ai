# realtrust frontend — Application Skill

Implement and evolve the realtrust ai web frontend so it stays aligned with the backend API, transaction state machine, and journey milestones. The backend is the source of truth for legality and authorization; the frontend MUST pass RLS context (user/org/role) and MUST NOT rely on client-side logic for security.

## Non-negotiables (frontend spine)

- **Backend is authority**: Legality, state transitions, and visibility are enforced by the backend (PostgreSQL RLS). The frontend MUST NOT enforce authorization; it only passes identity and reflects what the API returns.
- **RLS headers on every request**: All API calls MUST send `X-User-Id`, `X-Organization-Id`, `X-Role` (and optional `X-License-State`, `X-Risk-Clearance`) from the current auth context. Dev uses seed-user switcher; production MUST use verified token-derived identity.
- **One identity per request**: The “current user” (and org/role) is set in AuthContext and used by the API client for every request. No client-supplied role in mutation bodies (e.g. transition role comes from session).
- **Journey-aligned UI**: Screens and navigation MUST align with the transaction macro-states and milestone facts. If the backend gates a transition on a milestone, the UI SHOULD reflect that (e.g. document checklist, CLOSED = deed + ownership transfer).
- **Types match API**: Frontend types in `src/types/api.ts` MUST stay in sync with backend Pydantic response/request schemas. API path strings MUST match backend router paths (see backend skill API surface).
- **No security in UI**: Hiding a button or nav item is UX only. The backend MUST reject unauthorized actions; the frontend MUST handle 403/404 and error codes (e.g. ILLEGAL_TRANSITION, PRECONDITION_FAILED) and surface them to the user.
- **Path rule (compliance with backend):** Backend routers **without** a prefix (offers, showings, documents, inspections, appraisals, events, escrow, title) expose paths as written (e.g. `transactions/{id}/offers`, `listings/{id}/showings`). Do NOT prepend the domain (e.g. use `transactions/${id}/offers`, not `offers/transactions/${id}/offers`). See `references/20-api-contract-frontend.md` and backend skill API surface table.

## Workflow

- **Adding a page or flow**: Ensure route and nav align with `21-ui-journey-and-pages.md`; add API calls in `src/lib/api.ts` with correct paths (check backend router); add or extend types in `src/types/api.ts`; use React Query hooks that pass `user` from `useAuth()`.
- **Changing API usage**: Update `api.ts` and `types/api.ts`; align paths with backend `api/v1/router.py` and endpoint decorators; handle new error codes in `ApiException`; surface errors to users via toast (`toastError`), not inline UI text.
- **Role-based UX**: Use `user.role` from `useAuth()` with **`src/lib/permissions.ts`** for all role-based visibility (create property/listing/transaction, upload property images, set cover, publish/unpublish listing, schedule showing, add showing feedback, order appraisal). Gates are UX only; backend RLS is the authority.
- **State machine in UI**: Use `TransactionState` and transition actions only as allowed by backend; display state badges and timeline from API data; do not infer legality in the client.

## Backend alignment

- **API base**: Backend serves at `API_V1_PREFIX` (e.g. `/realtrust-ai/v1`). Frontend `buildApiUrl(path)` prepends `NEXT_PUBLIC_API_BASE + API_V1_PREFIX`. Paths MUST match backend router: e.g. list offers = `transactions/${transactionId}/offers` (not `offers/transactions/...`). Full path table: `references/20-api-contract-frontend.md`.
- **Error codes**: Backend returns `detail.error.code` (ILLEGAL_TRANSITION, PRECONDITION_FAILED, NOT_FOUND, FORBIDDEN_BY_POLICY, etc.). Frontend `ApiException` exposes `code` and `details`; `getApiErrorMessage` maps codes and prefers server message when present (FORBIDDEN_BY_POLICY, PRECONDITION_FAILED). UI surfaces messages via toast.
- **REGULATED escrow/title**: Backend restricts funding confirm, disbursement, deed recording, ownership transfer to ESCROW_OFFICER with state gating (CLEAR_TO_CLOSE/CLOSED). Unauthorized or wrong-state requests return 403 FORBIDDEN_BY_POLICY with a server message; frontend toast shows that message. Funding/disbursement/deed/ownership lists may be empty for non-ESCROW (and non-LENDER for funding) — UX only; backend RLS is authority.
- **Transaction states**: Same enum as backend (`05-transaction-state-machine-spec`); CLOSED = deed recorded + ownership transfer confirmed.
- **Compliance (FINRA/SOC2 alignment):** Authorization is backend-enforced (RLS); frontend passes RLS headers on every request and never sends role/user_id in mutation bodies for role-scoped actions (e.g. transitions). Role-based nav/UX is for clarity only; backend denies unauthorized access. Keeping paths and types in sync with backend supports auditability and self-documentation.
